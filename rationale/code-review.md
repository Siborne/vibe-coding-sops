# 为什么需要代码评审

## 触发场景

一个 dev 花了两周实现功能，直接合入 main。一周后线上出 bug，没人知道那段逻辑是怎么写的，因为只有作者读过。更糟的是，其他三个模块也用了同样的模式，但写法和这个都不一样——各写各的。

## 代码评审的四个价值

### 1. 知识分散

没有评审的代码 = 作者是唯一的"知情者"。当作者休假、离职、或忘记了——代码就变成了黑盒。评审确保至少两个人理解每一行逻辑。

### 2. 隐性假设暴露

新人或不同领域的开发者会问"这里为什么这样做"，这些问题恰好暴露了代码中未记录、未简化的假设。作者在团队里待久了会产生"这还用说吗"的盲区，外部视角打破它。

### 3. 模式收敛

没有评审，每个人按自己的风格写。三周后，同样的功能有三种实现。评审时发现"这个我们之前讨论过，用 X 模式更好"，团队代码库才会趋向一致。

### 4. 学习加速

评审让每个人接触到代码库的不同角落，看到别人的解题思路。对于 vibe coding，评审还能帮助识别 AI 生成代码中的次优模式。

## 为什么安全预检放在七原则之前

安全漏洞和代码风格缺陷性质不同。风格缺陷在评审中可以讨论协商，但注入漏洞、硬编码密钥、权限缺失在合并前必须解决，没有妥协空间。

把安全预检放在七原则之前有两个作用：

1. **不可跳过** — 前置的检查关卡在人因上更显眼，不会被淹没在后续的评审讨论中。AI 必须先输出自检结果，然后才能进入下一阶段。
2. **输出可见** — 自检清单的格式化输出提供了可审计的证据。未来复查时，可以直接看到每次 AI 代码修改经过了哪些安全门槛。

A07（认证失效）和 A09（日志不足）标注为 [suggestion] 而非 [blocking]，是因为这两项涉及更广泛的系统级设计判断，不是纯代码模式可以判定。即便如此，AI 仍然需要逐项回答，不能跳过。

## 在 vibe coding 场景下更关键

AI 产出的代码有其特征：逻辑逻辑上通顺、格式化完美、但可能在边界条件、安全、并发、资源泄漏等方面存在隐患。代码评审是发现这些隐患的关防——不是不信任 AI，而是 AI 生成速度太快，人工审查必须跟紧。

## 没有这条规则的后果

- 同一份代码里出现三种风格的错误处理，维护者苦不堪言
- 关键安全/并发缺陷因为没有第二双眼睛而被遗漏
- 团队知识孤岛——每个人只知道自己的模块
- 代码质量缓慢但不可逆地下滑
