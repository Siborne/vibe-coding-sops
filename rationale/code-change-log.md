# 为什么需要代码变更记录

## 触发场景

你接手一个半年前维护过的项目，线上出了 bug。你翻 git log，看到一串 "fix"、"update"、"change stuff" 的提交，完全不知道当时改了什么、为什么改。你只能从头读代码，浪费几个小时还原上下文。

这就是没有变更记录的真实代价。

## 在 vibe coding 场景下更关键

AI 辅助编码的产出速度远超手写，一天可能产生几十处修改。这些修改的质量参差不齐——有的经过深思熟虑，有的是一次性试出来的。如果不记录，两周后连你自己都分不清哪些修改是有意为之、哪些只是 prompt 的副作用。

## 为什么必须是结构化记录而非零散注释

- **可索引**：放在固定目录下，按日期 + 主题命名，随时能找到
- **可追溯**：记录了修改前后的代码对比，不需要从 git diff 反推
- **根因分析**：强制要求写清"为什么"，避免拍脑袋修 bug 引入新问题
- **知识传递**：如果作者休假或离职，接手者不需要反向工程

## 没有这条规则的后果

- 同一个 bug 被反复修复又反复引入
- 重构时破坏历史修复点，因为不知道那个看起来多余的 `if` 到底是干什么的
- 排查线上问题只能靠 `git blame` + 猜
