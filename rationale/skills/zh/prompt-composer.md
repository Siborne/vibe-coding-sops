# 为什么需要 Prompt Composer

## 触发场景

用户想做一个用户认证系统，打开 Agent 窗口直接输入："帮我实现登录、注册、找回密码、OAuth 集成"。Agent 开始写代码，写了 500 行后发现方向偏了——用户想要 JWT，Agent 用了 Session；用户想要 GitHub OAuth，Agent 做了 Google。一个窗口里边改边骂，来回 20 轮才勉强对齐。

同一个需求，另一个用户用了两个窗口：先在 Ask 窗口把需求拆成 4 步 prompt，再逐个贴到 Agent 窗口执行。每步产出确认正确后再下一步，总共 6 轮对话完成，没有一次方向性返工。

## 核心问题

单个 Agent 窗口里，用户的需求描述通常是模糊的、跳跃的、不完整的。Agent 默认"先做再说"，在信息不足时猜测用户意图。结果：代码写得快，方向错得更快。

把"澄清需求"和"执行需求"拆成两个阶段，本质上是把**分析**和**实现**解耦了。第一个阶段只做一件事——搞清楚到底要做什么。第二个阶段只做一件事——按清楚的指令执行。

## 为什么必须是一个独立 skill 而非靠用户自觉

**用户不会自动拆分。** 大部分用户打开 Agent 窗口就开始描述需求，Agent 开始写代码，节奏已经起来了。没有人会在中途说"等等，我们先把 prompt 写好再继续"。Skill 的作用就是给这个拆分行为一个明确的入口和一套固定流程。

**拆分需要方法论。** 用户自己拆，典型做法是"第一步做什么、第二步做什么"，每步一句话。但好的 prompt 需要包含上下文、输入输出定义、约束条件。Skill 把这种写作方法固化下来，不依赖用户的 prompt 工程经验。

**Ask 模式天然适合做这个。** Ask 模式不写代码，只对话——这正好是澄清需求的最佳环境。Ask 没有"先做再说"的惯性，AI 会追问、会往回看、会反复确认。

## 为什么输出格式是"对话脚本"而非一个完整文档

- **执行窗口每次处理一件事**：一次给所有步骤，Agent 会跳着做或漏做。拆成单步，每步有明确的开始和结束
- **中断后可恢复**：Step 2 出问题了，不用重新做 Step 1。确认每个 Step 的产出正确，是给下一步打地基
- **角色定义放在最前面**：执行窗口 AI 先读角色再读任务，比"在 prompt 第一句写你是 xxx"更有效——因为角色定义了全程行为边界，不只是单步指令

## 没有这个 skill 的后果

- 复杂任务在单个 Agent 窗口里反复翻车，用户以为"AI 能力不够"，实际是 prompt 不够清楚
- 用户花大量时间在 Agent 窗口里澄清需求——Agent 窗口是为执行设计的，不是为澄清设计的
- 每个用户自己发明一套 prompt 写作方法，团队之间不通用
- 好的 prompt 写法停留在"经验"层面，无法复制给其他人
