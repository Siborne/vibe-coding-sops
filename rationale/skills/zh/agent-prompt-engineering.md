# 为什么需要 Agent Prompt Engineering

## 触发场景

一个开发者在构建智能客服 Agent 应用。他写了一个简单的 system prompt："你是客服，帮用户解决问题"，然后接到产品里。上线后发现：

- Agent 有时输出 500 字长文，有时只回"好的"，格式不统一导致前端渲染崩了
- 用户问"我的订单什么时候到"，Agent 编了一个日期（幻觉），因为 prompt 里没约束"不知道就说不知道"
- 需要 Agent 输出 JSON 给后端解析，但 10 次里有 3 次 JSON 格式不对，代码里 try-catch 写了一堆
- 复杂问题（同时涉及退款、投诉、物流）Agent 跳跃回答，漏掉了关键信息

开发者花了三天反复改 prompt，每次改一句话、测一下、不行再改。没有系统性方法，纯靠试错。

同一个需求，另一个开发者用了结构化提示词 + 数据结构转换 + CoT 的组合：角色定义清晰、输出 JSON schema 精确、复杂问题分步推理。第一版 prompt 上线，Agent 输出一致性 95%+，JSON 解析失败率 <1%。

## 核心问题

开发者给产品 Agent 写 prompt 时，习惯用"给人写指令"的方式——模糊、随意、依赖读者常识。但 LLM 不是人，它不会追问，信息不足时直接猜。结果：prompt 的模糊性直接转化为 Agent 产出的不可靠性，进而导致整个应用不稳定。

**Agent 应用开发中的 prompt 和 IDE 里驱动 AI 写代码的 prompt 根本不同：**

- IDE 里的 prompt：人在回路里，AI 写错了你能马上看到、立即纠正
- 产品里的 prompt：人不在回路里，Agent 输出直接暴露给终端用户或下游代码，错一次就是一次事故

这意味着产品 Agent 的 prompt 必须在"无人监督"的条件下也能稳定产出正确结果——这就是工程化提示词要解决的问题。

## 为什么必须是一个独立 skill 而非靠开发者自觉

**大多数开发者不知道 prompt 可以工程化。** "写 prompt"在很多人眼里就是写一段文字描述，不知道有 CoT、结构化模板、正反面示例这些工具。结果是 100 个开发者有 100 种写 prompt 的风格，质量全靠个人悟性。

**试错法成本远高于方法论。** 没有方法论的开发者改 prompt 是"改一句 → 测一下 → 不行 → 再改一句"。5 个技巧的组合可以让第一版 prompt 就接近可用基线，而不是从零碰运气。

**5 种技巧覆盖了 Agent 应用的典型失败模式：**

| 失败模式 | 对应技巧 |
|---------|---------|
| Agent 跳过推理直接猜答案 | CoT 思维链 |
| Agent 理解偏差，输出方向完全错误 | 结构化提示词 |
| 格式飘忽不定，下游解析失败 | 正反面示例引导 |
| 复杂问题遗漏关键信息 | 分治法 |
| 自然语言输出无法被代码消费 | 数据结构转换 |

## 为什么是这 5 种技巧，以及为什么按这个顺序

这 5 种技巧按**从简单到复杂、从通用到专用**排列：

1. **CoT 思维链** — 最基础的可靠性技术。让 Agent 展示推理过程，而不是直接输出答案。几乎所有需要推理的 Agent 都应该默认加 CoT。
2. **结构化提示词** — 最通用的框架。角色/任务/要求/格式四模块适用于 80% 的 Agent 场景。
3. **正反面示例引导** — 当结构化还不够精确时的加强手段。对比示范比单纯文字描述更直观。
4. **分治法** — 处理复杂度的武器。当单个 prompt 无法覆盖时使用。
5. **数据结构转换** — 和代码集成的桥梁。Agent 要想和软件系统对接，必须输出结构化数据。

这个顺序也是推荐的"增量使用"顺序：先从结构化打底，不够再加 CoT，还不够加示例，依次递进。不要一上来就五种全用。

## 没有这个 skill 的后果

- 开发者的产品 Agent prompt 质量全靠个人经验，同样的需求，A 写的 prompt 产出 95% 一致性，B 写的只有 60%
- Prompt 迭代依赖盲目试错，一个 prompt 改 30-50 轮才勉强可用
- Agent 输出不稳定 → 产品体验差 → 用户流失 → 团队认为"AI Agent 不靠谱"，砍掉项目
- 好的 prompt 工程经验停留在"口口相传"，无法沉淀为团队资产
- 跟 prompt-composer 混淆使用场景，把 Agent 应用 prompt 的设计逻辑套在 IDE prompt 上（或者反过来），效果更差
