# Why Branch & PR Workflow Rules Matter

## Trigger Scenario

A team just started using AI-assisted coding. Day one, three people's branch names are `login-fix`, `zhang/work`, and `feat/login`. Day three, someone merges `main` into a feature branch, producing a string of "Merge branch 'main' into feat/xxx" commits. PRs are opened with no description and no self-test record — the reviewer has no idea where to start. Branches aren't deleted after merge; a month later, `git branch -r` scrolls endlessly.

## Core Problem

AI won't manage Git for you. It has no instinct for "is this branch name right," "should I rebase now," or "did I self-test before opening a PR." Without explicit rules, under AI's high-output rhythm, Git history rapidly degrades — naming chaos, merge commit proliferation, branch accumulation, and PRs without context.

## Why "Experience" or "See What Happens" Doesn't Work

- Branch naming by experience → everyone's experience differs, producing five formats across ten branches
- PR size by feel → "feels small" but it's 3,000 lines — no one picks up the review
- Merge method by habit → some rebase, some merge — the history graph is a tangled mess
- Checklists by memory → something always gets forgotten

## Why It Matters More in Vibe Coding

AI-assisted coding dramatically increases branch creation frequency — a single session may open 3-5 branches. Without unified naming and cleanup rules:
- `git branch -a` shows 80 branches after three weeks, with no one knowing which are active
- Merge methods vary with each AI response — rebase one day, merge the next
- PR descriptions generated by AI but unconstrained — some absurdly detailed, some just "fix"

## Consequences Without This Rule

- Branch naming chaos — impossible to tell a branch's purpose from its name
- Git history graph becomes a "subway map" with more merge commits than feature commits
- Branches never cleaned up — the remote becomes a branch graveyard
- PRs without self-checklists — reviewers repeatedly flag basic issues
- Inconsistent merge methods make cherry-pick and rollback behavior unpredictable
