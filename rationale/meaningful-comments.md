# 为什么需要有意义注释

## 触发场景

你读到一段注释写着 `// validate input`，下面的代码有三层嵌套循环 + 正则 + 位运算。注释等于没说。你又读到一段没有任何注释的代码，一个奇怪的常数 `1492` 横在中间——是随手写的？是协议规定的？是 4 月调出来的最优值？不知道。

## 核心洞察：注释的敌人是"WHAT"

代码天然表达 WHAT——它是步骤的可执行描述。但代码无法表达：
- 为什么是这个步骤而不是另一个
- 这个步骤的边界条件是什么
- 之前尝试过什么、为什么不行
- 修改这段代码会破坏什么

注释应该填补这些空白。如果注释只是在翻译代码到自然语言（"遍历列表，过滤空值"对应 `[x for x in list if x]`），它就是噪音。

## 在 vibe coding 场景下更关键

AI 生成的代码往往"看起来很对"——逻辑通顺、命名规范、没有语法错误。但它会：
- 选择次优方案（因为训练数据偏向常见模式）
- 忽略项目特有的约束（因为不了解完整上下文）
- 制造看似合理的常数（实际上需要根据环境调整）

这些 AI 代码的"隐性瑕疵"必须通过注释暴露出来，否则过段时间谁都看不出来。

## 没有这条规则的后果

- "为什么不用 X"型失误：后来者把特意避开的方案当成"遗漏"补上，搞垮整个功能
- 常数被当作可调参数随意修改
- 重构时去掉"承重细节"，正确性悄然崩塌
