# 为什么需要代码风格声明

## 触发场景

一个新项目，AI 生成第一段代码。它用 Python 写了段逻辑，缩进 4 空格，类型注解时有时无，错误处理混用 try-except 和返回 None。第二天 AI 又生成了另一段，缩进变成了 2 空格，用 `raise` 而不是 `return None`。三天后，同一个文件里四种风格共存。

## 核心问题

AI 的训练数据覆盖所有风格，它不是你的团队。同一个 prompt 问两次，可能拿到两种不同风格的代码。如果不预先声明风格偏好，AI 会在每次回复中随机选择一个它见过的风格——一天内产出的代码就能写成"四种作者，四种习惯"。

## 在 vibe coding 场景下更关键

手写代码时，一个人的代码风格自然一致——因为只有一个脑子。AI 辅助编码时，"作者"切换了无数次——每次 AI 的回复都可能来自不同的风格区域。风格声明本质上是一个全局约束，把"每次回复"变成"所有回复"统一到一个标准下。

## 为什么不能默认靠"看现有代码推断"

- 存量代码可能本身就风格不一致（还没做声明的时候写的）
- 有些偏好是策略层面的（用不用策略模式），从代码中无法直接推断
- "禁止项"从代码中完全看不到——你能看到的是 `catch (Exception e) {}` 已经在那里了

## 没有这条规则的后果

- 同一个项目不同文件的代码风格互不一致
- 团队成员在 PR 里反复争论风格问题而非逻辑问题
- 重构时 bar 越高——不一致的代码 base 产生不一致的新代码
