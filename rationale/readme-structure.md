# 为什么 README 要按漏斗式组织

## 触发场景

你搜索到一个 GitHub 仓库，名字看起来相关。你打开 README，前 50 行是安装依赖列表、环境变量配置、Docker 启动命令。你滚动到底都没搞懂这东西到底干什么。你关掉标签页，换下一个。

大多数 README 的读者只会停留几秒到几十秒。

## 漏斗结构的心理学

漏斗结构匹配的是陌生人的决策流程：

1. **它做什么？** —— 这是"匹配"阶段。一句话 + 一张图，让读者判断"这能解决我的问题吗"。如果不能，省下双方时间。
2. **为什么在乎？** —— 这是"动机"阶段。同类工具这么多，凭什么花时间看这个？
3. **怎么用？** —— 这是"幻想"阶段。让人看到最终效果，建立"我想要这个"的冲动。
4. **怎么装？** —— 这是"行动"阶段。只有在决定投入后，安装步骤才有价值。

反过来就是大多数平庸 README：先塞一堆安装配置，读者在不知道这东西值不值的情况下就被要求投入精力。大多数人不投入。

## 在 vibe coding 场景下更关键

AI 辅助项目通常产出快、迭代多。README 是你项目对外部的唯一发言人。如果 README 是反漏斗的，即使代码写得再好，也没人知道它存在。

## 没有这条规则的后果

- 项目 star 数始终是个位数（不是代码不好，是介绍不来）
- 潜在的贡献者和使用者错过这个项目
- 自己三个月后回来也看不懂从哪开始
