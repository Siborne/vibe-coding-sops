# 为什么提交信息要有完整上下文

## 触发场景

你运行 `git blame`，想搞清楚三年前的一行代码为什么存在。提交信息写着 "fix"。没有 issue 链接，没有问题描述，没有方案讨论。这一行现在还在用吗？还是那个 bug 早已不存在了？你不知道。你只能留着它，永远留着。

## 提交信息是时间机器

代码的每一次修改都对应一个决策时刻。那个时刻的上下文——问题现象、备选方案、取舍理由——如果不写在提交信息里，就永远消失了。代码会演进，但"为什么变成现在这样"只有在提交信息里。

代码告诉后人 WHAT，注释告诉后人 WHY（局部），而提交信息告诉后人 WHY（全局）——这次修改的背景和动机。

## 在 vibe coding 场景下更关键

AI 辅助编码产生的提交频率远高手写。一天几十个 commit 不稀奇。如果每个 commit 都是 "update"、"fix"，三天后的 `git log` 就是一本无字天书。你需要通过提交信息快速定位"哪个 commit 引入了哪个变更"。

## 没有这条规则的后果

- `git blame` 追到一行可疑代码，提交信息是 "refactor"，什么信息都没得到
- 回滚时不知道该回滚到哪个版本
- 合并冲突时无法判断哪个分支的修改更"正确"
- 新成员 onboard 时无法通过 git log 快速了解项目演进历史
