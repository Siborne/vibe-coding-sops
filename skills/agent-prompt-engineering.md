# Agent Prompt Engineering

构建 AI Agent 应用时，为应用内置 Agent 编写高质量提示词的 5 种工程化技巧。适用于在代码中通过 API 调用 LLM 的场景——你写的 prompt 会嵌入软件代码，成为产品的一部分。

> 区别于 prompt-composer（驱动 IDE 里的 AI 做开发），本 skill 关注的是**你开发的软件产品中的 Agent 提示词设计**。

## 何时使用

- 正在开发一个调用 LLM API 的 Agent 应用，需要设计 system/user prompt
- 产品中的 AI Agent 产出质量不稳定，需要系统性优化提示词
- 需要 Agent 输出特定格式（JSON、报告、分类结果），供下游代码消费
- 任务涉及多步推理，需要 Agent 展示可追溯的思考过程

## 5 种技巧

### 1. CoT 思维链 — 让 Agent 展示推理过程

**核心思路**：引导 Agent 逐步思考，将复杂问题分解为多个中间步骤，降低直接输出答案的错误率。

**适用场景**：数学计算、代码调试、逻辑推理、需要可解释结果的业务决策

**在你的应用代码中**：

```python
system_prompt = """你是一个数学辅导助教。回答问题时必须展示完整推理过程，
不要直接给出答案。"""

user_prompt = """请用逐步推理的方式解决以下问题：
如果 3 个苹果和 4 个香蕉共花费 38 元，2 个苹果和 5 个香蕉共花费 40 元，
求单个苹果和香蕉的价格。

请按以下步骤回答：
1. 设未知数并列出方程
2. 解方程组
3. 验证结果
4. 给出最终答案"""

response = client.chat.completions.create(
    model="deepseek-chat",
    messages=[
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": user_prompt}
    ],
    temperature=0.3
)
```

**原则**：
- 步骤要明确编号，每步产出可验证
- system prompt 里定调（"必须展示推理过程"），user prompt 里给具体步骤
- 简单任务不需要 CoT — 增加不必要的 token 消耗和延迟

### 2. 结构化提示词 — 模块化设计减少歧义

**核心思路**：将提示拆分为角色、任务、要求、输出格式四个模块，让 Agent 精准理解意图。

**适用场景**：生成标准化报告、代码审查、需要特定格式输出的业务场景

**在你的应用代码中**：

```python
user_prompt = """【角色】你是一名资深市场分析师
【任务】生成 2023 年新能源汽车市场分析报告
【要求】
1. 数据对比：对比 2022 与 2023 年销量、市场份额（表格呈现）
2. 趋势预测：分析未来 3 年技术发展方向
3. 风险提示：评估政策与供应链风险
【输出格式】Markdown，包含标题、表格和分点列表"""

response = client.chat.completions.create(
    model="deepseek-chat",
    messages=[{"role": "user", "content": user_prompt}],
    temperature=0.3
)
```

**原则**：
- 角色要具体：不写"优秀的分析师"，写"擅长新能源汽车行业的资深市场分析师"
- 要求要可验证：不写"分析得好一点"，写"包含近 3 年销量数据的对比表格"
- 简单对话不需要结构化 — 过度包装浪费 token

### 3. 正反面示例引导 — 对比强化标准

**核心思路**：同时提供正确和错误示例并标注问题，让 Agent 通过对比理解期望标准。

**适用场景**：格式规范严格（邮件、客服回复、代码生成）、容易出重复性错误的场景

**在你的应用代码中**：

```python
user_prompt = """请根据以下正反示例，生成一封正式的会议邀请邮件：

【正确示例】
主题：关于项目进度评审会议的通知
尊敬的张经理：
您好！我们定于 2023 年 11 月 15 日下午 2 点，在公司会议室 A 召开项目进度评审会议。
会议议程包括：1. 各模块进度汇报 2. 风险项讨论 3. 下阶段计划。
请提前准备相关材料，准时参加。

【错误示例】
主题：开会
老张：
明天下午来开会，记得带资料。
（问题：主题不明确、称呼不正式、缺少时间地点和议程）

请按照正确示例的标准，为以下场景生成邮件：[场景描述]"""

response = client.chat.completions.create(
    model="deepseek-chat",
    messages=[{"role": "user", "content": user_prompt}],
    temperature=0.3
)
```

**原则**：
- 错误示例必须标注"为什么错"，裸贴一个坏例子没有引导效果
- 正确示例展示完整的理想输出形态
- 1-2 个正反对比即可，不需要穷举
- 正反面示例可以直接从产品历史数据中提取

### 4. 分治法 — 复杂任务拆解为子问题

**核心思路**：将复杂任务分解为相对独立的子任务，按逻辑顺序依次解决，最后整合结论。

**适用场景**：多维度分析、需要从多个角度综合判断的业务场景

**在你的应用代码中**：

```python
user_prompt = """请分步骤分析电商平台用户流失的主要原因：

第一步：用户行为分析
- 统计流失用户的最后操作路径
- 分析用户停留时间分布

第二步：产品体验评估
- 检查主要功能页面的转化率
- 收集用户反馈中的高频问题

第三步：竞品对比
- 比较主要竞争对手的核心指标
- 分析竞品的优势功能

请按以上三个步骤依次分析，最后给出整合结论。"""

response = client.chat.completions.create(
    model="deepseek-chat",
    messages=[{"role": "user", "content": user_prompt}],
    temperature=0.4
)
```

**原则**：
- 子任务之间要相对独立，减少耦合
- 每步子任务进一步细化为 2-3 个具体分析点
- 最后必须有一个"整合结论"步骤，把子结果汇总
- 子任务产出可以分别校验，出问题时能定位到哪一步

### 5. 数据结构转换 — 自然语言转结构化数据

**核心思路**：明确定义目标数据结构的字段和类型，让 Agent 将非结构化输入转换为 JSON/XML 等标准化格式。

**适用场景**：信息提取、数据清洗、自然语言转 API payload、非结构化文档解析

**在你的应用代码中**：

```python
user_prompt = """将以下产品描述转换为 JSON 格式：

产品名称：智能温控水杯
特点：
- 容量：450ml
- 材质：食品级 304 不锈钢
- 温度显示：LED 数显屏
- 保温时长：12 小时（热水）/ 24 小时（冷水）
价格：299 元

要求 JSON 结构：
{
  "product_name": "string",
  "specs": {
    "capacity": "string",
    "material": "string",
    "features": ["string"],
    "price": "number"
  }
}

要求：
- 价格字段为 number 类型，不含"元"字
- 容量中的数字和单位分开存储"""

response = client.chat.completions.create(
    model="deepseek-chat",
    messages=[{"role": "user", "content": user_prompt}],
    temperature=0.2  # 结构化输出建议用较低温度
)
```

**原则**：
- 结构定义要完整，避免遗漏关键字段
- 每个字段明确类型约束（string / number / boolean / array）
- 可选字段必须显式标注为 optional
- 数字、日期等特殊格式要给出示例
- 结构化输出建议 temperature 设为 0.2 或更低
- 先在简单案例上验证结构定义合理，再逐步增加复杂度

## 技巧选择指南

| 你的 Agent 需要做什么 | 推荐技巧 | 为什么 |
|---------------------|---------|--------|
| 推理计算、需要可解释的结果 | CoT 思维链 | 分步推理降低错误率，过程可追溯 |
| 输出格式固定、角色边界清晰 | 结构化提示词 | 模块化减少歧义，输出更可控 |
| 格式规范严格、容易出重复性错误 | 正反面示例引导 | 对比学习比单纯指令更直观 |
| 多维度综合分析 | 分治法 | 拆解降低复杂度，每步独立验证 |
| 从自然语言中提取结构化数据 | 数据结构转换 | 类型约束避免输出格式随机 |
| 以上多种需求叠加 | 组合使用（见下方） | 单一技巧不足以覆盖复杂场景 |

## 组合使用

复杂 Agent 的提示词通常需要组合多种技巧：

- **结构化 + CoT**：用结构化模板定义角色和输出格式，在"要求"中嵌入 CoT 分步推理
- **分治 + 数据结构转换**：每个子步骤输出结构化 JSON，最后一步整合为完整结果
- **正反面 + 结构化**：在结构化模板的"要求"中引用正反面示例
- **CoT + 数据结构转换**：先让 Agent 推理判断，再将判断结果转成结构化数据

## 常见错误

| 错误 | 改正 |
|------|------|
| system prompt 写"你是一个优秀的 AI 助手" | 写具体角色和技术边界："你是电商客服，只能回答订单和退货相关问题" |
| 要求写"回复质量高一点" | 写可验证约束："拒绝回答时给出替代建议，每条回复不超过 200 字" |
| 简单分类也用结构化模板 + CoT | 简单的分类/判断直接写一句话 prompt，不要过度设计 |
| 分治法子任务之间有强依赖 | 确保每个子任务可以独立求解，依赖关系显式标注 |
| JSON 结构不标字段类型 | 每个字段明确 string/number/boolean/array |
| temperature 用默认值 | 结构化输出建议 0.1-0.3，创意任务建议 0.6-0.8 |
| 把 prompt 写死在代码里 | 至少抽成常量或配置文件，方便 A/B 测试和迭代 |

## 与应用代码的集成建议

- **Prompt 版本管理**：提示词应像代码一样做版本管理，变更时在 commit message 中记录原因和效果
- **A/B 测试**：关键 prompt 设计两个版本，线上对比效果后再确定最终版
- **异常处理**：Agent 输出的 JSON 解析失败时，给模型一次重试机会（在 prompt 中附上解析错误信息）
- **Token 预算**：给你的 Agent prompt 设定 token 上限，控制每次调用的成本

## 原则

- **不猜测**：对模型能力边界不确定时，先写最小 prompt 测试，不要在 prompt 里"碰运气"
- **可验证**：每个要求和约束都要能在代码中对 Agent 输出做自动化校验
- **YAGNI**：prompt 覆盖当前业务需求即可，不擅自加"最好也做 X"
- **先简后繁**：先用最简单的技巧（结构化），不够再加 CoT 或分治
- **Token 意识**：prompt 长度是成本，简单任务不要过度工程化
